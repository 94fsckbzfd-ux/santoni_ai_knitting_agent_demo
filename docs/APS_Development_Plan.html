APS 待开发计划

面向 Athena / APS 研发协同的产品需求草案：把 APS 从排单效率工具升级为利润率、交期确定性和工厂经营决策工具。

关联产品：Athena 关联模块：APS Adapter / ROI Engine / Planner Copilot 更新日期：2026-05-26

1. APS 当前定位

APS 是应用在生产计划排单步骤的网页版工具。客户 ERP 接到订单，采购完成物料采购、库管完成原材料出库后，订单进入计划人员排单环节。APS 自动接收 ERP 订单信息，支持计划人员排单和插单，并生成工单下发到机台进行排产。

上游ERP 订单、物料状态、采购/出库完成情况。
核心用户计划人员、生产主管、工厂老板、IT/系统管理员。
核心动作排单、插单、机台分配、工单生成、排产下发。
Athena 价值解释排单、评估 ROI、识别风险、推动跟进动作。
产品叙事建议：APS 不只解决“排单人员少做事”，更要帮助老板提高利润率、减少交期风险、提升机器小时收益。

2. 现场问题与产品机会

现场问题 风险 产品机会
客户自研 ERP 通过 API 主动向 APS 派单；ERP 端口重启或 bug 修复时，APS 可能丢失工单。 计划人员不信任 APS 数据完整性，需要人工补单或反复核对。 建立订单同步防丢机制和同步健康看板。
APS 提高排单效率，降低排单人员工作量，但排单人员未必有动力用 APS 替代自己的经验流程。 系统被视为替代人工的工具，实际 adoption 低。 把 APS 做成 Planner Copilot，帮助排单人员解释、比较和确认方案。
客户设备联网程度不高，APS 的自动化优势无法充分发挥。 计划和实际脱节，排程优化难以闭环。 支持低联网模式，先用人工确认、扫码、批量导入等方式产生价值。
GTM 叙事偏“降本”，但老板更关心订单利润、交期风险和机台资源回报。 客户老板层付费意愿不足。 与 Athena ROI Engine 联动，做利润率导向排单和插单影响模拟。

3. 待开发计划优先级

优先级 需求 开发内容 验收指标
P0 ERP-APS 订单同步可靠性 ERP push + APS pull 双通道；订单 ACK；失败重试；幂等 order_id；订单版本号；异常队列；同步健康看板；手动 reconciliation。 端口重启后不漏单；APS 能显示 ERP 与 APS 订单差异；异常订单可追踪、可恢复。
P0 排单健康看板 显示最近同步时间、待排订单、已排订单、异常订单、ERP 拉取状态、工单下发状态、机台待确认状态。 计划人员能在一个页面确认“今天 APS 数据是否可信”。
P1 Planner Copilot 生成 2-3 个排单方案；解释每个方案的交期、换款、机台占用、瓶颈和风险；支持人工调整、锁定、确认；记录人工修改原因。 排单人员把 APS 视为助手，而不是替代者；人工改排原因可沉淀。
P1 插单影响模拟 插单后显示受影响订单、延期风险、换款/换纱/调机时间、机台占用变化、可能损失的机器小时收益。 计划人员和老板能判断“这个急单值不值得插”。
P2 利润率导向排单 接入 Athena ROI Engine，结合订单金额、成本、产能、交期、换款、良率和机会成本，输出高利润订单优先级和低价值占产能预警。 老板能看到 APS 对利润率、机器小时收益和交付确定性的影响。
P2 针织行业专用约束模型 机器型号、针数、机号、筒径、纱线可得性、款式/颜色/组织切换成本、技工技能、维护窗口、良率和返工风险。 排单建议更贴近无缝针织真实生产约束。
P3 低联网部署模式与实际闭环 手动状态回填、扫码更新工单进度、班组长批量确认、每日产量导入、联网成熟度评分、计划 vs 实际偏差分析。 客户设备未完全联网时，APS 仍可落地并逐步积累数据。
P3 自学习排程优化 沉淀计划工时 vs 实际工时、换款时间偏差、停机原因、延期原因、人工改排原因，用于优化未来排程参数。 APS 排程参数能从真实生产结果中持续校准。

4. Athena 联动设计

APS Adapter

负责把 APS 的订单、排单方案、机台资源、工单状态、异常同步事件结构化暴露给 Athena。

  • get_orders() 获取待排/已排/异常订单。
  • get_schedule() 获取当前排程和机台占用。
  • simulate_insert_order() 模拟插单影响。
  • confirm_schedule() 人工确认后下发排程。
ROI Engine

负责把 APS 的产能、交期、换款和插单影响转化为老板可理解的经营指标。

  • 机器小时收益。
  • 订单贡献毛利。
  • 插单机会成本。
  • 低利润订单占用瓶颈机台预警。

Agent 输出示例

问题 Athena 应输出
这个急单要不要插？ 可插/不建议插；受影响订单；延期风险；损失机器小时收益；建议加价或协商交期。
今天 APS 数据可信吗？ ERP 最近同步时间；异常订单数；疑似漏单；需要手动 reconciliation 的订单。
为什么 APS 这样排？ 瓶颈机台、换款成本、交期优先级、纱线到料状态、人工/班次约束。
这批订单利润率能不能提高？ 高利润订单优先级、低价值占产能订单、可合并生产订单、建议换机/换时段方案。

5. 需要 APS 研发确认的问题

  1. 当前 ERP → APS 的 API 是否已有唯一订单号、版本号和更新时间字段？
  2. APS 是否可以主动按时间窗口向 ERP 拉取订单增量和全量对账？
  3. 工单下发到机台后，APS 能否接收机台或人工回填的执行状态？
  4. 当前排程算法是否支持有限产能、换款时间、机台兼容性和人工班次约束？
  5. 是否已有插单模拟能力？如果没有，最小可行版本需要哪些字段？
  6. 排单人员人工调整排程后，是否能记录修改原因和修改前后差异？
  7. APS 是否可以为 Athena 提供只读 API，用于 ROI 测算和老板看板？

6. GTM 话术建议

旧说法 新说法
APS 可以减少排单人员工作量。 APS + Athena 帮老板判断哪些订单更赚钱，哪些插单会损失利润。
APS 可以提高排单效率。 APS 可以让交期风险、瓶颈机台和插单影响变得可见。
APS 需要设备联网才能发挥作用。 APS 可以先用低联网模式落地，再逐步推动机台联网和数据闭环。
对老板层的核心价值表达：提高利润率、减少错误接单、避免低价值订单占用关键产能、提高交付确定性。

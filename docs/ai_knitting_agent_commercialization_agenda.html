Internal Discussion Agenda

AI Knitting Agent 商业化落地方案讨论

本材料用于内部讨论 AI Knitting Agent 从免费入口、3 个月 MVP、付费模块、试点客户到长期商业化路径的落地方案。

讨论目标 形成 MVP 测试与试点共识
时间窗口 3 个月后推出 MVP
首批客户 现有大客户与关系好的 OEM/ODM 工厂
核心商业模式 SaaS + 模块包 + 积分

一、已形成的基本判断

AI Knitting Agent 应被定位为面向针织产业链的智能生产与运营 Agent，而不是单一 AI 设计工具。它的长期价值在于连接设计、打样、排产、设备维护、工单、机器数据与订单 ROI，让工厂和品牌方在效率、确定性和经营决策上获得可衡量提升。

AI 设计打样 Arachne Knit Pilot / SWS Factory Intelligence Package APS 排单优化 设备 Knowledge Base Smartex 视觉验布 订单 ROI 概念验证 Advanced Maintenance Package

Smartex 最终合作方案

Smartex 智能视觉 AI 验布仍是 Smart Machine 的重要组成部分。最终方案是：Smartex 提供 BOM，由 Santoni 在国内采购、组装和测试，避免元器件中国采购、葡萄牙组装后再发回中国造成的关税和物流成本；Smartex 提供软件算法，并提供 30 小时免费 support，超过后按每台已销售机器 750 欧收取服务费。

Belotti 会议新增判断：SWS / Arachne

SWS 或 Arachne 的最终方向不是设计师与设计软件的交互，而是互联网海量数据与设计软件的交互。目标是让 AI 基于海量数据，通过 SWS 生成可以投产的服装图片，再让消费者自行选择。

Belotti 会议新增判断：Athena Video

Athena Video 不能只追求真实，还要表达 company vision 和 Athena MVP 的参照物。视频应让公司内部和客户理解 Santoni 想成为什么，以及 3 个月 MVP 应该对齐什么样的产品原型。

二、目标客户与关键角色

OEM / ODM 工厂

第一阶段商业化重点客户。尤其适合 50 台以上机器、有打样、排产、插单和设备管理复杂度的工厂。

  • 老板 / 管理层：关注订单 ROI、利润、开机率、交付确定性。
  • IT 部门：负责项目管理、系统落地、账号和积分分配。
  • 打样师：减少与设计师沟通时间，快速出样。
  • 排单人员：提升排单、插单效率和机器利用率。
  • 挡车工 / 车间技术工：快速查询设备问题和操作建议。
  • 机修团队：处理工单、预警机器和维护建议。

品牌方

品牌方不是第一优先级，但可以作为设计打样能力的外部客户与生态入口。

  • 设计师团队：快速生成服装原型，查看仿真渲染效果。
  • 供应链团队：更清楚地向工厂表达需求，减少样衣沟通轮次。
  • 采购 / 管理层：如果愿意采购，可放宽使用边界。

三、3 个月 MVP 建议范围

第 1 阶段：免费入口

  • Web 版无需登录即可使用。
  • 接入设备相关 Knowledge Base。
  • 支持现场技术人员快速问答。
  • 支持基础问题诊断与工单创建。
  • Arachne Knit Pilot 作为自然入口，引导客户创建账号。

第 2 阶段：设计打样 MVP

  • 基于已有 SWS 和升级中的 Arachne Knit Pilot。
  • 支持设计原型生成。
  • 支持款式说明、打样沟通材料和基础工艺建议。
  • 逐步加入仿真与渲染图生成。
  • 主要面向 OEM/ODM 工厂，品牌方可销售。
  • 长期方向是互联网数据驱动 SWS/Arachne 生成可投产服装，而不只是设计师与软件对话。

第 3 阶段：工厂智能包试点

  • APS 不独立销售，纳入 Factory Intelligence Package。
  • 验证排单优化、插单响应和机器资源匹配。
  • 订单 ROI 先作为概念验证，不在 MVP 中过度承诺。
  • 面向 50 台以上机器的工厂优先试点。

四、产品包与收费逻辑

收费设计原则

保留“基础版、进阶版、专业版”作为客户容易理解的套餐包装，但内部定价应从“功能堆叠”转向“价值指标”。每个产品包只突出一个主要计费单位，再用账号、机器或积分作为辅助扩展项，避免客户同时面对版本、账号数、机器数和积分四套逻辑。

五、订阅版本与积分机制

基础版

  • 适合轻量试用、小团队和从免费版转化的客户。
  • 包含基础账号、少量免费积分和基础 Arachne Knit Pilot 功能。
  • 不建议绑定复杂机器额度，避免客户理解成本过高。

进阶版

  • 适合有明确打样和协作需求的工厂。
  • 提供更多账号、团队协作、项目保存和更高积分额度。
  • 包含更完整的 AI 设计打样能力。

专业版

  • 适合 50 台以上机器的中大型工厂。
  • 包含 Factory Intelligence Package。
  • 建议以年费和项目实施为主，未来加入订单 ROI、老板驾驶舱和高级报表。

Anonymous Mode

  • 无需登录即可使用。
  • 开放基础设备 Knowledge Base 问答和基础 Arachne Knit Pilot 体验。
  • 可做限速、防滥用和基础安全控制。
  • 不保存项目、不绑定团队、不承诺历史记录。

Account Mode

  • 创建账户后可保存设计、工单、项目和积分余额。
  • 公司管理员统一购买和分配积分。
  • 支持团队协作、权限管理和销售线索沉淀。
  • 适合作为免费版向付费版转化的关键节点。

积分机制建议

积分同时绑定公司账户和个人账户。公司账户拥有总积分池，客户方管理员将积分分配给设计师、打样师、排单人员、IT、机修团队等个人账户。积分应主要用于高成本、高价值、客户主动触发的动作；普通问答、基础知识库查询和基础工单流转不建议扣积分。

建议消耗积分的复杂交互 原因
完整服装设计方案、批量款式方案、仿真渲染 计算成本高，且直接影响客户商业效率。
SWS / Arachne Knit Pilot 高级工艺生成 属于高价值专业能力，应与免费问答区分。
APS 排产优化、插单影响模拟 涉及复杂算法和工厂生产决策。
订单 ROI 分析、老板报表 直接对应管理层付费意愿。
远程诊断报告、机器健康分析、月度设备运行报告 适合纳入 Advanced Maintenance Package 或高阶积分场景。

六、收费风险与修正建议

1. 避免收费维度过多

基础版、进阶版、专业版是外部包装；内部报价要明确每个包的主计费单位。设计打样按席位，工厂智能按工厂和机器，高级维护按已接入机器，复杂 AI 动作按积分。

2. 积分只管复杂交互

积分不应覆盖所有使用行为，否则客户会感觉被限制。普通问答和基础服务免费，高价值动作才消耗积分，客户才能理解“为什么这个动作要付费”。

3. APS 不独立零售，但必须显性计价

APS 可以打包进入 Factory Intelligence Package，但不能在销售表达里变成免费赠品。建议明确它是该包的核心能力，并在报价说明中体现价值。具体研发需求见 APS 待开发计划。

4. Pilot Fee 优于完全免费试点

涉及排产、数据接入和工厂流程调研的试点，建议收取 Pilot Fee，并约定签年度订阅时可部分或全部抵扣。这样可以筛选认真客户，也能保护内部交付资源。

5. ROI 先做 Beta，不做强承诺

订单 ROI 应先作为老板驾驶舱和 ROI Lite 访谈项目，不建议在 MVP 阶段承诺完整算法。收费前必须确认客户能提供哪些订单、成本、排产和质量数据。

6. Smart Machine 改造单独收费

未来竞争对手设备接入或老设备改造，不应混入 SaaS 年费。建议拆成硬件 / 传感器改造费、安装实施费和每台机器每月监控服务费。

7. Smartex Support Fee 单独建模

Smartex 不再按每台设备固定收费，而是提供 30 小时免费 support；超过免费支持时长后，按每台已销售机器 750 欧收取服务费。需要明确 support 计时方式、触发条件、软件算法更新、售后责任和缺陷识别责任边界。

七、订单 ROI 算法建议方向

订单 ROI 目前应作为产品力方向和老板层销售故事，不建议在 3 个月 MVP 中承诺完整算法。建议先通过内部讨论和 OEM 工厂老板访谈，确认真正影响接单决策的数据口径。

ROI Lite 数据需求

  • 订单金额、数量、单价和交期。
  • 用纱类型、用纱量和纱线单价。
  • 预计生产时间、机器类型和机器数量。
  • 人工成本、打样成本和编程成本。
  • 当前排产队列和机器空闲情况。
  • 预计毛利、交付风险和是否值得接单。

ROI Pro 数据需求

  • 机器实际 OEE / 开机率。
  • 换款、换纱、调机时间。
  • 历史返工率、次品率和质量风险。
  • 停机概率、机修风险和备件风险。
  • 客户付款周期、复购可能性和战略价值。
  • 接单与不接单的产能机会成本。

八、试点销售路径

首批试点客户

  • 现有 Santoni 大客户。
  • 关系好的 OEM/ODM 工厂。
  • 优先选择 50 台以上机器、有真实排产和打样痛点的客户。
  • 品牌方如有明确采购意愿，可作为设计打样方向试点。

销售价值故事

  • 短期主打设计打样提效。
  • 中期主打 Factory Intelligence Package。
  • 长期主打老板能看清订单 ROI。
  • Factory Intelligence 试点建议收取 Pilot Fee，签年度合同时可抵扣。
  • 未来可考虑面向竞争对手设备较多的工厂提供 Smart Machine 改造服务。
老板层最容易感知的指标 建议表达方式
打样周期缩短时间 从“几天出样”变成“更快形成可讨论方案和可执行工艺方向”。
开机率提升 减少等待、空转、排产冲突和异常停机带来的产能损失。
插单响应速度 让插单影响可见，快速判断是否值得调整现有排产。
每月节约人工 减少重复沟通、人工排单、人工查资料和手工汇总报表。
订单利润更清楚 把经验判断转化成可记录、可比较、可复盘的数据决策。

九、建议新增讨论点

1. MVP 的成功标准

3 个月后的测试不能只看功能是否上线，还要定义客户是否愿意继续使用、愿意创建账号、愿意提供数据和愿意付费试点。

2. 数据授权与边界

工厂数据涉及订单、成本、排产和设备状态，需要提前定义哪些数据必须采集、哪些只做本地化处理、哪些可匿名化用于算法优化。

3. Arachne Knit Pilot 与 Agent 的入口关系

需要明确客户第一眼看到的是 Arachne Knit Pilot、AI Knitting Agent，还是二者融合后的统一入口。

4. 内部责任分工

研发、应用、售后、销售、IT 和产品管理需要明确谁负责知识库、谁负责客户试点、谁负责数据接口、谁负责商业报价。

5. 与 Smart Machine 的节奏配合

Advanced Maintenance Package 很重要，但不应拖慢 3 个月 MVP。建议与传感器选型和数据采集系统并行推进。

6. 竞争对手设备接入策略

未来如果向竞争对手设备较多的工厂提供改造服务，需要讨论技术标准、服务边界、商业风险和销售话术。

7. 收费主指标与报价复杂度

需要确认每个模块到底按什么收费：设计打样按席位，工厂智能按工厂和机器，高级维护按接入机器，复杂 AI 交互按积分。

8. 试点收费策略

需要讨论哪些试点免费、哪些试点收 Pilot Fee，以及 Pilot Fee 在后续年度订阅中如何抵扣。

9. Smartex 商业与责任边界

需要确认 Santoni 根据 Smartex BOM 在国内采购、组装和测试后，Smartex 在软件算法、30 小时免费 support、超时服务费、算法更新和验布效果上的责任边界。

10. SWS / Arachne 数据驱动路线

需要讨论互联网数据、趋势数据、消费者选择和 SWS/Arachne 可投产设计之间的产品闭环，以及这条路线与 Arachne Knit Pilot 入口的关系。

11. Athena Video v2 叙事方向

视频需要同时承担 company vision 和 MVP reference 两个任务：既展示 Santoni 想做什么，也展示 Athena MVP 应该参照什么样的可用产品形态。

十、内部会议建议 Agenda

  1. 确认产品定位
    AI Knitting Agent 是免费工具、设计工具、设备服务，还是工厂智能化平台？建议统一为“针织产业链智能生产与运营 Agent”。
  2. 确认 3 个月 MVP 范围
    是否同意 MVP 聚焦免费 Web Agent、设备 Knowledge Base、基础工单、Arachne Knit Pilot 入口和 AI 设计打样，而不把 Advanced Maintenance 作为首要交付。
  3. 确认免费版与积分边界
    Anonymous Mode 和 Account Mode 如何划分？普通问答是否永久免费？哪些交互定义为复杂交互？积分是否按公司账户统一购买并由管理员分配？
  4. 确认套餐结构与主计费单位
    基础版、进阶版、专业版分别包含哪些功能？设计打样、工厂智能、高级维护和品牌访问分别按什么主指标收费？超出后如何收费？
  5. 确认试点收费方式
    哪些试点可以免费换案例？哪些试点需要收 Pilot Fee？如果客户后续签年度订阅，Pilot Fee 是否抵扣？
  6. 确认 Smartex 智能视觉验布商业分工
    是否采用 Smartex 提供 BOM、Santoni 国内采购组装测试、Smartex 提供软件算法和 30 小时免费 support、超时按每台已销售机器 750 欧收服务费的模式？售后和责任边界如何写入协议？
  7. 确认 SWS / Arachne 的长期产品叙事
    是否将其定义为互联网数据与设计软件之间的智能生成引擎，而不是单纯的设计师交互工具？
  8. 确认 Athena Video v2 目标
    视频是否以 company vision 和 Athena MVP reference 为主线，而不是只追求真实场景演示？
  9. 确认试点客户名单
    从现有 Santoni 大客户和关系好的 OEM/ODM 工厂中筛选 3 到 5 家试点客户，明确联系人、场景和可开放数据。
  10. 确认 ROI 概念验证方式
    是否先访谈 OEM 工厂老板，收集接单决策、成本结构、排产风险和利润判断逻辑，再定义 ROI Lite。
  11. 确认内部资源与负责人
    明确产品、研发、销售、售后、应用工程师和 IT 的责任边界，以及每两周的里程碑检查机制。
  12. 确认从 Strategy 到 Execution 的治理机制
    Belotti 的战略判断需要被拆解成产品路线图、MVP 范围、任务 Owner、里程碑和验收标准，避免 CEO 直接深入每个执行细节。

十二、Strategy-to-Execution 治理风险

新增组织判断

Belotti 对 AI Knitting Agent、Smart Machine、SWS/Arachne 和 Athena MVP 的战略方向已经比较完整，但执行层缺少将战略拆解成计划、任务、优先级、责任人和验收标准的机制。结果是 Belotti 需要直接从细节上盯 MVP、视频、产品逻辑和具体事务，这会降低管理效率，也会让商业化负责人被迫前移去承担产品经理职责。

核心风险

  • 战略停留在高层认知里，无法稳定转化成执行路线图。
  • MVP 没有清晰 owner、范围、里程碑和验收标准。
  • CEO 被迫进入视频、功能、优先级等执行细节。
  • 商业化负责人因为没有可销售 MVP，被迫承担产品经理角色。
  • DT、产品、研发、商业化之间责任边界不清晰。

建议治理机制

  • 建立一页式 Product Roadmap，将战略拆成 3 个月 MVP、6 个月试点、12 个月商业化版本。
  • 为每个模块指定 DRI，包含 Athena、Arachne/SWS、APS、Smartex、Knowledge Base 和商业化。
  • 每两周进行一次 MVP review，只讨论进度、阻塞、决策和下一步。
  • 每个模块必须有验收标准，例如视频 brief、MVP demo、客户试点、报价矩阵和数据接口。
  • 商业化负责人应负责市场定位、客户验证、试点设计、收费模型和 GTM，不应默认承担产品交付 owner。
角色 建议职责边界
Belotti 定义愿景、战略优先级、关键取舍和最终商业判断，不应长期承担执行项目经理角色。
Danny / DT 负责技术组织、产品交付节奏、研发资源协调和 MVP 执行机制。
产品 Owner / PM 负责把战略拆成 PRD、roadmap、任务拆解、验收标准和跨团队推进。目前这是明显缺口。
商业化负责人 负责客户画像、价值主张、定价、试点客户、销售材料和商业反馈，不应成为默认产品经理。

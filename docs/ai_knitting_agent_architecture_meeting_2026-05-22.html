AI Knitting Agent 架构沟通稿

面向技术团队的讨论材料：统一说明 AI Knitting Agent 的目标、入口设计、部署架构、核心能力，以及未来需要接入的内外部系统。

会议日期：2026-05-22 讨论对象：技术团队 / 子系统负责人 核心原则：多个入口，共用一个 Agent Core

0. 一句话结论

AI Knitting Agent 不应该被定义成一个单独聊天机器人，而应该是连接品牌设计、Arachne Knit Pilot 参数化定制、SWS 高级打样、OEM 工厂服务和 Santoni 内部系统的智能工作层。

业务价值 缩短从设计意图到无缝打样、生产文件、订单 ROI 判断、行业机会识别和服务响应之间的沟通链路。
技术原则 SWS、Web、Arachne Knit Pilot、飞书只是不同入口，底层应复用同一个 Agent Core 和工具接口。
当前策略 先稳定 Agent Core、状态机、工具契约、Guardrails，再考虑 Hermes 和生产部署。
部署判断 开发阶段可本地跑；内部试用可放公司服务器；外部访问和正式 Bot 需要公网 HTTPS 或公司网关。

1. 做 AI Knitting Agent 的目标和目的

1.1 行业背景

针织行业的 value chain 从品牌方、OEM/ODM 工厂、纱线供应商延伸到原材料。品牌方设计师在设计衣服时，通常先根据用途和功能判断使用无缝工艺还是布料缝合工艺。如果选择无缝工艺，设计稿需要进一步转化为无缝织机可以识别的打样程序。

当前痛点在于：设计师或 ODM 先通过 Adobe Illustrator、CLO 3D 等软件形成效果图，再交给工厂打样工程师。打样工程师需要使用 Diagraph、i-Polaris 或 SWS 等软件，把设计图、纱线型号、机器能力和工艺限制转化成打样程序。这个过程中需要不断判断设计是否可织、样品和设计图会有什么差异，并反复沟通修改。

1.2 Agent 要解决的问题

  • 减少设计师和打样师之间关于设计可行性、纱线、组织结构、机器参数和样衣差异的反复沟通。
  • 帮助品牌方或 ODM 更早判断设计图能否转化为无缝工艺，以及转化后可能出现的差异。
  • 把 SWS、Arachne Knit Pilot、服务案例库、机器数据、工单系统等子系统通过统一 Agent Core 串起来。
  • 帮助 OEM 工厂老板判断订单是否值得接、是否有能力接，以及接单后的 ROI、利润和产能风险。
  • 定期监测行业、竞品、客户、政策和海外市场动态，并转化为销售、产品、服务和市场动作建议。
  • 让客户服务场景从“人工理解问题”转向“Agent 先收集信息、匹配案例、尝试安全在线排查、再决定是否派工”。
  • 沉淀 Santoni 的工艺知识、服务经验和工具使用流程，形成可复用的数字化能力。

2.1 Hermes 的位置

Hermes 可以作为 Agent Runtime，而不是替代 Santoni 自己的业务 Agent Core。建议第一阶段让 Hermes 通过 HTTP API 或 MCP tool 调用当前 Agent Core；等状态机、工具契约和 Guardrails 稳定后，再逐步把 SOP、技能和工具注册到 Hermes。

推荐路线：先做 Hermes → Santoni Agent Core 适配，不要一开始就把 Designer / Service 业务逻辑全部重写进 Hermes。

2.2 部署阶段

阶段 1：个人电脑开发

验证 Agent Core、Hermes 适配、API schema、工具调用逻辑。只适合开发者本机使用。

阶段 2：公司内网服务器

团队连入公司网络或 VPN 后访问。适合内部 demo、服务团队测试、SWS/Arachne Knit Pilot 联调。

阶段 3：Staging 云环境

用于飞书 webhook、外部客户试用、公网 HTTPS、日志和权限验证。

阶段 4：生产环境

接入真实工单、客户数据、敏感工具、审计、监控、备份和高可用。

不建议现在直接把所有能力部署到 VPS 并开放给外部客户。应先完成接口契约、权限、Guardrails、测试基线和敏感工具审计。

3.3 Demo 策略调整：Arachne Knit Pilot-first，SWS Advanced

当前 demo 不必强依赖 SWS。Arachne Knit Pilot 已经具备客户选择款式、花纹、图案、颜色并生成无缝织机生产文件的能力；如果进一步把 Arachne Knit Pilot 内所有可变参数设计成清晰 schema，Agent 就可以用自然语言定义这些参数，并在用户交互层实现 3D 静态图的呈现和修改。

因此建议当前设计侧 demo 采用 Arachne Knit Pilot-first 路线：先用 Arachne Knit Pilot 承载款式设计、纱线选择、面料结构、颜色/图案配置、3D 静态展示和生产文件生成。未来 SWS 成熟后，再作为 SWS Advanced Package 引入更丰富的参数、更强仿真和专业打样能力。

这样可以降低 demo 依赖风险：现在用 Arachne Knit Pilot 做可见、可交互、可出文件的闭环；未来用 SWS 增强专业打样和高阶商业化套餐。

3.4 新增产品力算法：OEM 订单 ROI / 接单决策

这是面向 OEM 工厂老板和经营管理层的重要功能。Agent 不只回答“这个产品怎么做”，还要回答“这个订单值不值得接、有没有能力接、接了能产生多少效益”。该能力建议设计成可审计的 ROI Engine，而不是完全依赖 LLM 自由计算。

核心输入

订单数量、目标价格、交期、产品参数、工艺难度、机器型号与产能、纱线成本、人工成本、能耗、换款时间、预计良率、当前产能占用、付款条款。

核心输出

是否建议接单、预计 ROI、毛利/贡献毛利、机器小时收益、产能瓶颈、交期风险、建议报价、可接受条件和风险解释。

技术实现上建议采用“规则/公式/数据模型 + Agent 解释层”：算法服务负责确定性计算，Agent 负责收集缺失参数、解释假设、生成 what-if 对比，例如“如果交期延长 5 天，ROI 是否改善”“如果换用另一种纱线，毛利会变多少”。

3.4.1 APS 排单与利润率提升

APS 是客户 ERP 接到订单、采购和库管完成物料准备后进入生产计划排单的网页版工具。原则上计划人员不需要打开 ERP，APS 自动接收 ERP 订单信息，支持排单和插单，并生成工单下发到机台进行排产。

详细待开发需求已经整理到 APS_Development_Plan.html，包括 ERP-APS 同步防丢、排单健康看板、Planner Copilot、插单影响模拟、利润率导向排单和低联网部署模式。
现场问题 产品/研发建议 Agent 价值
ERP 通过 API 主动向 APS 派单；ERP 端口重启或 bug 修复时，APS 可能丢失工单。 建立 ERP push + APS pull 双通道，增加订单 ACK、重试、幂等 order_id、版本号、异常队列和同步健康看板。 Agent 可以监测“疑似漏单/同步异常”，提醒计划人员并触发 APS 向 ERP 重新拉取。
APS 降低排单人员工作量，但排单人员可能缺少替代自己经验流程的动力。 设计 Planner Copilot：解释排单建议、展示插单影响、对比多个方案，并保留人工最终确认权。 Agent 把 APS 定位为排单人员助手，而不是替代者，提高使用意愿。
客户设备联网程度不高，APS 的自动化优势无法充分发挥。 支持低联网部署模式：手动状态回填、扫码/批量导入、机台状态人工确认和联网成熟度评分。 Agent 可先在半自动场景产生价值，并逐步推动设备联网。
GTM 叙事应从“APS 降低排单人工”升级为“APS + Athena 提高利润率”：帮助老板识别高利润订单、减少低价值插单、降低交期风险、提升机器小时收益，并把排单结果反馈给 ROI Engine。

3.4.2 Santoni IOT System / 中国本地化 Nautilus

Santoni IOT System 是中国团队本地化的机联网系统，可以理解为本地化 Nautilus。它不是 APS 的替代品，而是机器联网和实际执行数据层：APS 负责计划排单和插单模拟，IOT 负责机台状态、订单执行、报警、停机、OEE、产量、质量和现场资源数据回传。

核心能力

大圆机、无缝/袜机实时监控；订单监控、订单管理、款式管理、Send Order、独立 Weave Part Order 和 knitting program 下发；Dashboard；Production、Efficiency、Stop Event、OEE、Fabric Detect、Payroll、AI Analysis 等数据分析；API；工厂资源配置。

Athena 价值

为 Service Agent 提供报警和停机上下文；为 APS 提供计划 vs 实际反馈；为 ROI Engine 提供机器小时收益、实际产能、良率和停机成本；为 Advanced Maintenance 和 Smart Machine 提供健康评分、故障预警和质量分析数据。

数据域 关键数据 接入建议
机台与现场资源 Machine Code、Machine SN、Machine Model、Plant、Employee、Shift、Shift Arrange、Fabric Settings。 作为 Installed Base 和客户现场上下文，支撑服务、排产和 ROI。
生产与效率 Order Code、Style Code、Ideal/Actual Output、Production Time、Down Time、Performance、Availability、Quality、OEE。 通过只读 API 进入 Athena，用于 ROI、APS 实际执行反馈和管理报表。
报警与停机 Fault Code、Fault Description、Fault Start/End Time、Fault Duration、Stop Count、Time To Repair、Missing Record Duration。 接入 Service Agent 的故障诊断和相似案例检索；未来可转成维护工单。
质量与验布 Defect Count、Defect Type/Detail、Roll Code、Defect Picture、Recovery Time、Fabric Type。 连接 Fabric KB、质量缺陷库和 Smartex / Smart Machine 视觉验布能力。
订单/程序下发 Send Order、Weave Part Order、Knitting Program Dispatch。 属于高风险写操作，必须通过权限、审计、人工确认和客户现场授权控制。
架构原则：第一阶段建议 Athena 以只读方式接入 IOT 数据，优先用于诊断、分析和建议；涉及订单下发、程序下发或机器控制的动作，必须作为受控工具单独设计权限和审批流程。

3.4.3 PMO Agent / Project Intelligence Agent

PMO Agent 是 Athena 的内部项目管理型子 Agent，不面向客户销售。它解决的问题是：当 Athena 子模块越来越多时，项目进度不能依赖人工主观填写，也不能让 LLM 自由判断，而要基于任务、验收 checklist、文档 Change Log、代码/PR、测试结果和会议决策等证据自动计算。

核心架构

Project Evidence Adapter 读取飞书任务、Git/PR、CI、部署记录、文档 Change Log、会议纪要和项目 checklist；Project Graph 维护模块、owner、依赖、交付物、验收项、风险和决策关系；Progress Calculation Engine 用确定性规则计算进度、延期和风险。

Agent 价值

PMO Agent 负责生成项目周报、风险预警、需要 Danny / Belotti 或跨部门确认的决策清单、owner 提醒和下一步建议。LLM 只做总结和解释，不直接“感觉”项目完成百分比。

输入证据 计算逻辑 输出
任务、checklist、PR、测试、文档 Change Log、会议纪要、试点反馈。 按模块权重计算：需求定义、数据/API 准备、开发、测试验收、文档交付、决策完成度。 自动进度百分比、延期状态、风险等级、缺失证据、决策需求和周报。
建议落地路径：第一阶段先让项目进度页基于 checklist 自动计算；第二阶段导入飞书任务、Git/PR 和文档 Change Log；第三阶段做定时 PMO Agent，自动生成周报、风险预警和会议议程。

3.5 新增子 Agent：Industry Radar 行业情报雷达

Industry Radar Agent 的定位不是新闻推送，而是“行业信号 → Santoni 影响 → 可执行动作”的情报分析系统。它应定期检索国内外高可信来源，并把信息转化为销售、产品、市场、服务和管理层可以执行的建议。

定时任务流水线
Source Registry维护来源、关键词、地区、可信度、抓取频率。
Scheduled Collector每日/每周/紧急触发抓取网页、RSS、新闻和搜索结果。
Normalizer去重、抽取日期、公司、地区、产品、政策和证据 URL。
Relevance Scorer按 Santoni 相关性、客户影响、竞品威胁、商机价值打分。
Insight Writer生成三层输出：发生了什么、为什么重要、建议做什么。
Delivery & Tasks推送到 Web/飞书/邮件，并可转成 CRM 或销售跟进任务。
建议定时任务

每日晨报：工作日 08:30。每周深度报告：每周一。展会日历：每周更新并在展前 90/30/7 天提醒。竞品/政策预警：高优先级来源每 2-4 小时检查。

建议输出字段

标题、分类、地区、来源 URL、发布日期、可信度、影响对象、重要性评分、建议负责人、建议动作、跟进截止日期。

3.6 行业情报的检索方式

信息类别 检索方式 示例来源
展会日历 优先抓取官方展会网站、主办方新闻稿、参展商名单和论坛议程。 ITMA、ITMA ASIA + CITME、Texprocess、Techtextil
竞品动态 抓取竞品官网新闻、产品页、展会新闻、LinkedIn/YouTube 官方账号和经销商公告。 Santoni、Lonati、Mayer & Cie、Shima Seiki、Stoll、KARL MAYER、Reifenhäuser 等官方来源。
客户侧动态 抓取品牌官网、可持续报告、投资者关系、供应链公告、材料创新新闻。 Nike Sustainability、Decathlon、Adidas、Uniqlo、Shein、Temu、Lululemon 等官方来源。
政策与贸易风险 监测官方政策、贸易数据库、法规更新、化学品限制、碳足迹和 ESG 规则。 WTO Textiles、EU Textiles Strategy、EU PFAS
海外市场机会 组合检索地区 + textile investment / garment export / factory expansion / policy incentive / machinery import 等关键词。 印度、越南、孟加拉、巴基斯坦、印尼、埃及、土耳其等政府、行业协会、园区和财经媒体。
技术与材料趋势 检索专利、论文、展会论坛、品牌材料新闻、供应商新品和行业媒体。 AI 工艺生成、数字孪生、视觉质检、预测性维护、节能电控、再生纤维、生物基材料、功能纱线。
Guardrail：所有情报必须保留来源 URL 和发布日期；未经二次确认的传闻只能标记为“待验证”，不能作为事实推送。政策、贸易和投资建议应标明“不构成法律或投资建议”。

4. 入口设计与目标客户

四类入口的定位应该清晰区分：Arachne Knit Pilot 面向当前可落地的客户自助定制和参数化设计，SWS 面向未来高级专业打样，Web 面向轻量交互和 demo，飞书面向内部协作。

入口 目标用户 典型任务 特点
SWS 入口 品牌设计师、ODM 设计师、OEM 打样师、Santoni 应用工程师 设计修改、文件转换、可织性判断、纱线纹理仿真、打样程序生成、版本差异解释。 未来高级专业入口。适合在 SWS 成熟后承载更复杂参数、更强仿真和更专业的打样程序修改。
Arachne Knit Pilot 入口 品牌客户、小批量定制客户、电商/定制业务客户、OEM 业务团队 选择款式、纱线、面料结构、花纹、图案、颜色，获得搭配建议和生产可行性提示，查看 3D 静态图并一键生成生产文件。 当前 demo 的优先设计入口。Agent 可做智能导购、参数定义助手、3D 修改助手、生产文件生成前检查。
Web 入口 潜在客户、销售 demo 对象、非 SWS 用户、客户设备工程师 轻量设计咨询、上传参考图、服务报修、案例排查、订单 ROI 测算、测试 Agent 能力。 低门槛入口。支持登录模式和免登录模式；登录模式保留项目历史和订单测算记录，免登录模式每次新会话。
飞书 Bot Santoni 内部服务、销售、项目、管理团队 工单通知、派工审批、项目状态查询、客户问题摘要、订单 ROI 摘要、每日服务风险提醒。 协作入口，不适合复杂 3D 或图形编辑。重点是任务流、通知、审批和状态同步。
Industry Radar 看板 / 推送 Santoni 销售、产品、市场、区域负责人、管理层 查看每日/每周行业情报、竞品预警、展会日历、海外市场机会、政策风险和商机任务。 不作为独立客户入口，而是内部决策入口。可嵌入 Web 后台、飞书推送或 CRM follow-up。

4.1 登录与匿名使用

登录模式

适合 SWS、Arachne Knit Pilot、Web 的正式用户。可关联客户、项目、历史文件、机器、权限和长期上下文。

免登录模式

适合销售 demo、首次试用、轻量咨询。每次都是新的交互过程，不保留长期记忆和敏感信息。

对外表达时建议使用“账号体系记录身份、项目历史和授权信息”，避免使用“记录账号密码”这类说法。密码应由认证系统处理，Agent 不保存明文密码。

5. 给各子系统开发负责人的接口准备清单

每个子系统不需要一开始就完整接入，但需要尽早确认可被 Agent 调用的最小接口、权限边界和数据格式。

准备项 建议说明
API / Tool Contract 明确输入 schema、输出 schema、错误码、超时、幂等规则和版本号。
权限模型 明确哪些角色可以读数据、创建任务、修改数据、执行高风险工具。
人工确认 派工、生成激活密码、提交生产文件、修改真实工单等动作必须定义确认流程。
审计日志 记录谁在什么入口、什么 session、调用了什么工具、使用了什么输入、得到什么结果。
测试数据 提供 mock 数据、脱敏真实案例和 golden conversation，用于回归测试。
数据边界 定义哪些数据可以进入 Agent 长期记忆，哪些只能短期使用，哪些绝不能保存。
数据库接入方式 CRM(SQL)、CPQ(主要数据在 MongoDB)、Service System(MySQL) 应优先通过各自业务 API 或 Adapter 暴露能力；如需直连数据库，先从只读账号、白名单字段和审计日志开始。
失败策略 外部系统不可用时，Agent 应明确告知用户、保留上下文，并提供人工处理或稍后重试选项。

5.1 接入前建议完成的技术准备

  1. 修复中文编码和中文关键词，确保中英文路由稳定。
  2. 把当前隐式 session state 改成显式 workflow state。
  3. 补正式 Guardrails：设计生成边界、服务安全边界、敏感工具边界。
  4. 将 Service Case、Activation Password、SWS、Arachne Knit Pilot、CRM、CPQ、Service System、OEM ROI Engine、Industry Radar、PMO Agent 等能力抽象为标准工具接口。
  5. 新增待办：整理 SWS chain file 的 AI 协助开发方案。chain file 本质是控制无缝机初始化、送纱、吹气等动作的 motion command sequence，短期不让 LLM 自由生成原始命令，而是先建设 chain file 样本库、motion command 语义说明、chain parser/validator、可复用 chain module library、SDI/四色图/chain 映射关系、规则校验和工程师审核流程，用于实现 AI-assisted chain programming。
  6. 建立回归测试：设计入口、服务入口、Arachne Knit Pilot 定制入口、订单 ROI 测算、行业情报摘要、锁机激活、派工确认、异常场景。
  7. 确认部署策略：开发本机、公司内网服务器、staging 云环境、生产环境分别承担什么职责。

6. 建议会议讨论结论

  • 确认 AI Knitting Agent 是统一 Agent Core，而不是 SWS Agent、Arachne Knit Pilot Agent、Web Agent、飞书 Agent 四套独立系统。
  • 确认第一阶段优先做 Web + Service Case + Arachne Knit Pilot 参数化设计闭环，用于快速验证业务闭环。
  • 确认 Arachne Knit Pilot 是当前 demo 的优先设计入口；SWS 在未来成熟后作为 Advanced Package，承载更强仿真、更多参数和专业打样能力。
  • 确认 OEM 订单 ROI / 接单决策是新增核心产品力算法，应作为可审计的算法服务接入 Agent，而不是单纯让 LLM 做财务估算。
  • 确认 PMO Agent 是内部项目管理型子 Agent，用证据和规则自动计算 Athena 项目进度、延期状态、风险和决策需求。
  • 确认 Industry Radar 是新的内部决策型子 Agent，通过定时任务把行业信息转化成 Santoni 的销售、产品、市场和服务动作建议。
  • 确认 Hermes 可作为运行时和工具调度层，但不替代 Santoni 自己的业务知识、状态机和工具规则。
  • 确认每个子系统负责人需要提供可调用接口、权限边界、测试数据和异常策略。

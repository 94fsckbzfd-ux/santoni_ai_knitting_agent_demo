AI Knitting Agent 架构沟通稿

面向技术团队的讨论材料：统一说明 AI Knitting Agent 的目标、入口设计、部署架构、核心能力，以及未来需要接入的内外部系统。

会议日期：2026-05-22 讨论对象：技术团队 / 子系统负责人 核心原则：多个入口，共用一个 Agent Core

0. 一句话结论

AI Knitting Agent 不应该被定义成一个单独聊天机器人，而应该是连接品牌设计、MyKnit 参数化定制、SWS 高级打样、OEM 工厂服务和 Santoni 内部系统的智能工作层。

业务价值 缩短从设计意图到无缝打样、生产文件、订单 ROI 判断和服务响应之间的沟通链路。
技术原则 SWS、Web、MyKnit、飞书只是不同入口，底层应复用同一个 Agent Core 和工具接口。
当前策略 先稳定 Agent Core、状态机、工具契约、Guardrails，再考虑 Hermes 和生产部署。
部署判断 开发阶段可本地跑；内部试用可放公司服务器；外部访问和正式 Bot 需要公网 HTTPS 或公司网关。

1. 做 AI Knitting Agent 的目标和目的

1.1 行业背景

针织行业的 value chain 从品牌方、OEM/ODM 工厂、纱线供应商延伸到原材料。品牌方设计师在设计衣服时，通常先根据用途和功能判断使用无缝工艺还是布料缝合工艺。如果选择无缝工艺，设计稿需要进一步转化为无缝织机可以识别的打样程序。

当前痛点在于：设计师或 ODM 先通过 Adobe Illustrator、CLO 3D 等软件形成效果图，再交给工厂打样工程师。打样工程师需要使用 Diagraph、i-Polaris 或 SWS 等软件，把设计图、纱线型号、机器能力和工艺限制转化成打样程序。这个过程中需要不断判断设计是否可织、样品和设计图会有什么差异，并反复沟通修改。

1.2 Agent 要解决的问题

  • 减少设计师和打样师之间关于设计可行性、纱线、组织结构、机器参数和样衣差异的反复沟通。
  • 帮助品牌方或 ODM 更早判断设计图能否转化为无缝工艺，以及转化后可能出现的差异。
  • 把 SWS、MyKnit、服务案例库、机器数据、工单系统等子系统通过统一 Agent Core 串起来。
  • 帮助 OEM 工厂老板判断订单是否值得接、是否有能力接，以及接单后的 ROI、利润和产能风险。
  • 让客户服务场景从“人工理解问题”转向“Agent 先收集信息、匹配案例、尝试安全在线排查、再决定是否派工”。
  • 沉淀 Santoni 的工艺知识、服务经验和工具使用流程，形成可复用的数字化能力。

1.3 面向三类核心业务场景

场景 核心用户 Agent 目标 业务输出
设计 / 打样 品牌设计师、ODM 设计团队、OEM 打样工程师、SWS 用户 把设计意图、参考图、纱线和机器能力转化为可执行的无缝打样方向。 可行性判断、纱线/组织/机器建议、SWS 打样程序方向、仿真差异说明、修改建议。
服务 / 售后 客户设备工程师、Santoni 服务工程师、服务经理 快速理解设备问题，优先安全在线排查，必要时创建工单并派工。 服务 intake、案例匹配、在线排查步骤、备件建议、工单摘要、工程师派工建议。
订单 ROI / 接单决策 OEM 工厂老板、业务负责人、计划负责人、报价负责人 评估订单是否值得接、产能是否能承接、资源瓶颈在哪里、接单后预计收益和风险。 ROI、毛利/贡献毛利、机器小时收益、产能占用、交期风险、建议报价或接单条件。

2. Agent 架构和部署

建议架构不是“每个入口各做一个 Agent”，而是“多个入口 + 统一 Agent Gateway + 统一 Agent Core + 标准工具适配层”。

2.1 Hermes 的位置

Hermes 可以作为 Agent Runtime，而不是替代 Santoni 自己的业务 Agent Core。建议第一阶段让 Hermes 通过 HTTP API 或 MCP tool 调用当前 Agent Core；等状态机、工具契约和 Guardrails 稳定后，再逐步把 SOP、技能和工具注册到 Hermes。

推荐路线：先做 Hermes → Santoni Agent Core 适配，不要一开始就把 Designer / Service 业务逻辑全部重写进 Hermes。

2.2 部署阶段

阶段 1：个人电脑开发

验证 Agent Core、Hermes 适配、API schema、工具调用逻辑。只适合开发者本机使用。

阶段 2：公司内网服务器

团队连入公司网络或 VPN 后访问。适合内部 demo、服务团队测试、SWS/MyKnit 联调。

阶段 3：Staging 云环境

用于飞书 webhook、外部客户试用、公网 HTTPS、日志和权限验证。

阶段 4：生产环境

接入真实工单、客户数据、敏感工具、审计、监控、备份和高可用。

不建议现在直接把所有能力部署到 VPS 并开放给外部客户。应先完成接口契约、权限、Guardrails、测试基线和敏感工具审计。

3.1 计划具备的核心功能

3.2 计划接入的内部和外部系统

系统 系统类型 Agent 接入价值 建议阶段
SWS 新一代无缝打样软件 / 未来高级包 未来提供更完整的设计文件转打样程序、设计修改、纱线纹理级仿真、AI 交互设计和高级参数能力。 P2
MyKnit 客户自助定制 Web 应用 客户选择款式、花纹、图案、颜色后，一键生成无缝织机生产文件；当前 demo 可优先使用 MyKnit 覆盖部分 SWS 能力，包括款式设计、纱线选择、面料结构和 3D 静态展示。 P0

ERP / MES / 成本数据 生产与财务数据源 提供机器负荷、班次、历史产量、人工成本、能耗、换款时间、良率、实际成本和交付记录。 P2
Service Case Library 服务知识库 沉淀历史故障、在线排查步骤、安全警告、派工触发条件。 P0
Service System 售后服务系统 / MySQL 创建工单、派工、状态跟踪、维修闭环、客户反馈。 P1
锁机激活密码平台 受控工具系统 处理锁屏、租赁状态、激活密码生成。需要严格权限、日志和人工确认。 P1 高风险
机器主数据 客户/设备数据 确认客户、机器型号、序列号、安装历史、保修/租赁状态。 P1
纱线库 材料数据 用于设计可行性、仿真纹理、成本、性能和供应可得性判断。 P2
APS / 排产系统 生产计划系统 估算打样周期、产能、交期、生产资源。 P2
DPP / 产品数据系统 产品生命周期数据 沉淀产品开发资料、材料信息、工艺数据和版本记录。 P2
设备数据 / IoT 机器遥测 读取报警、运行状态、停机、历史故障、关键参数。 P2
备件 / 库存系统 供应链数据 判断备件库存、仓库位置、随工携带可能性。 P2
飞书 / 邮件 / 企业微信 协作通知 审批、派工通知、状态提醒、日报总结。 P1
数据库类型用于帮助技术团队评估 Adapter 方案。Agent 层不应直接暴露 SQL、MongoDB 或 MySQL 的生产库写权限；推荐由各子系统提供受控 API，或先建立只读/受限权限的 Adapter 服务。

3.3 Demo 策略调整：MyKnit-first，SWS Advanced

当前 demo 不必强依赖 SWS。MyKnit 已经具备客户选择款式、花纹、图案、颜色并生成无缝织机生产文件的能力；如果进一步把 MyKnit 内所有可变参数设计成清晰 schema，Agent 就可以用自然语言定义这些参数，并在用户交互层实现 3D 静态图的呈现和修改。

因此建议当前设计侧 demo 采用 MyKnit-first 路线：先用 MyKnit 承载款式设计、纱线选择、面料结构、颜色/图案配置、3D 静态展示和生产文件生成。未来 SWS 成熟后，再作为 SWS Advanced Package 引入更丰富的参数、更强仿真和专业打样能力。

这样可以降低 demo 依赖风险：现在用 MyKnit 做可见、可交互、可出文件的闭环；未来用 SWS 增强专业打样和高阶商业化套餐。

3.4 新增产品力算法：OEM 订单 ROI / 接单决策

这是面向 OEM 工厂老板和经营管理层的重要功能。Agent 不只回答“这个产品怎么做”，还要回答“这个订单值不值得接、有没有能力接、接了能产生多少效益”。该能力建议设计成可审计的 ROI Engine，而不是完全依赖 LLM 自由计算。

核心输入

订单数量、目标价格、交期、产品参数、工艺难度、机器型号与产能、纱线成本、人工成本、能耗、换款时间、预计良率、当前产能占用、付款条款。

核心输出

是否建议接单、预计 ROI、毛利/贡献毛利、机器小时收益、产能瓶颈、交期风险、建议报价、可接受条件和风险解释。

技术实现上建议采用“规则/公式/数据模型 + Agent 解释层”：算法服务负责确定性计算，Agent 负责收集缺失参数、解释假设、生成 what-if 对比，例如“如果交期延长 5 天，ROI 是否改善”“如果换用另一种纱线，毛利会变多少”。

4. 入口设计与目标客户

四类入口的定位应该清晰区分：MyKnit 面向当前可落地的客户自助定制和参数化设计，SWS 面向未来高级专业打样，Web 面向轻量交互和 demo，飞书面向内部协作。

入口 目标用户 典型任务 特点
SWS 入口 品牌设计师、ODM 设计师、OEM 打样师、Santoni 应用工程师 设计修改、文件转换、可织性判断、纱线纹理仿真、打样程序生成、版本差异解释。 未来高级专业入口。适合在 SWS 成熟后承载更复杂参数、更强仿真和更专业的打样程序修改。
MyKnit 入口 品牌客户、小批量定制客户、电商/定制业务客户、OEM 业务团队 选择款式、纱线、面料结构、花纹、图案、颜色，获得搭配建议和生产可行性提示，查看 3D 静态图并一键生成生产文件。 当前 demo 的优先设计入口。Agent 可做智能导购、参数定义助手、3D 修改助手、生产文件生成前检查。
Web 入口 潜在客户、销售 demo 对象、非 SWS 用户、客户设备工程师 轻量设计咨询、上传参考图、服务报修、案例排查、订单 ROI 测算、测试 Agent 能力。 低门槛入口。支持登录模式和免登录模式；登录模式保留项目历史和订单测算记录，免登录模式每次新会话。
飞书 Bot Santoni 内部服务、销售、项目、管理团队 工单通知、派工审批、项目状态查询、客户问题摘要、订单 ROI 摘要、每日服务风险提醒。 协作入口，不适合复杂 3D 或图形编辑。重点是任务流、通知、审批和状态同步。

4.1 登录与匿名使用

登录模式

适合 SWS、MyKnit、Web 的正式用户。可关联客户、项目、历史文件、机器、权限和长期上下文。

免登录模式

适合销售 demo、首次试用、轻量咨询。每次都是新的交互过程，不保留长期记忆和敏感信息。

对外表达时建议使用“账号体系记录身份、项目历史和授权信息”，避免使用“记录账号密码”这类说法。密码应由认证系统处理，Agent 不保存明文密码。

5. 给各子系统开发负责人的接口准备清单

每个子系统不需要一开始就完整接入，但需要尽早确认可被 Agent 调用的最小接口、权限边界和数据格式。

准备项 建议说明
API / Tool Contract 明确输入 schema、输出 schema、错误码、超时、幂等规则和版本号。
权限模型 明确哪些角色可以读数据、创建任务、修改数据、执行高风险工具。
人工确认 派工、生成激活密码、提交生产文件、修改真实工单等动作必须定义确认流程。
审计日志 记录谁在什么入口、什么 session、调用了什么工具、使用了什么输入、得到什么结果。
测试数据 提供 mock 数据、脱敏真实案例和 golden conversation，用于回归测试。
数据边界 定义哪些数据可以进入 Agent 长期记忆，哪些只能短期使用，哪些绝不能保存。
数据库接入方式 CRM(SQL)、CPQ(主要数据在 MongoDB)、Service System(MySQL) 应优先通过各自业务 API 或 Adapter 暴露能力；如需直连数据库，先从只读账号、白名单字段和审计日志开始。
失败策略 外部系统不可用时，Agent 应明确告知用户、保留上下文，并提供人工处理或稍后重试选项。

5.1 接入前建议完成的技术准备

  1. 修复中文编码和中文关键词，确保中英文路由稳定。
  2. 把当前隐式 session state 改成显式 workflow state。
  3. 补正式 Guardrails：设计生成边界、服务安全边界、敏感工具边界。
  4. 将 Service Case、Activation Password、SWS、MyKnit、CRM、CPQ、Service System、OEM ROI Engine 等能力抽象为标准工具接口。
  5. 建立回归测试：设计入口、服务入口、MyKnit 定制入口、订单 ROI 测算、锁机激活、派工确认、异常场景。
  6. 确认部署策略：开发本机、公司内网服务器、staging 云环境、生产环境分别承担什么职责。

6. 建议会议讨论结论

  • 确认 AI Knitting Agent 是统一 Agent Core，而不是 SWS Agent、MyKnit Agent、Web Agent、飞书 Agent 四套独立系统。
  • 确认第一阶段优先做 Web + Service Case + MyKnit 参数化设计闭环，用于快速验证业务闭环。
  • 确认 MyKnit 是当前 demo 的优先设计入口；SWS 在未来成熟后作为 Advanced Package，承载更强仿真、更多参数和专业打样能力。
  • 确认 OEM 订单 ROI / 接单决策是新增核心产品力算法，应作为可审计的算法服务接入 Agent，而不是单纯让 LLM 做财务估算。
  • 确认 Hermes 可作为运行时和工具调度层，但不替代 Santoni 自己的业务知识、状态机和工具规则。
  • 确认每个子系统负责人需要提供可调用接口、权限边界、测试数据和异常策略。
